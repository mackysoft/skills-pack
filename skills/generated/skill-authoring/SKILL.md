# skill-authoring

## 目的
エージェントスキルを作成、更新、レビューし、必要な利用場面で意図した判断と作業が成立する状態にする。
スキルの成果は、責務と発火面が一致した `name` と `description`、作業に必要な本文と補助ファイル、挙動の検証結果である。

## 作業種別

| 作業 | 扱い |
| --- | --- |
| 作成 | 必要な能力を独立したスキルとして設計し、挙動検証まで行う。 |
| 更新 | 既存の責務を保ちながら、根拠のある変更を反映し、役割を失った記述と補助ファイルを整理する。 |
| レビュー | 作成、更新と同じ基準で評価し、問題、根拠、必要な差分を返す。修正も求められている場合は更新として進める。 |

## フロー

### Phase 1: 必要性と根拠を確定する
#### 根拠を集める
依頼、既存スキル、実際の利用記録、成果物、対象領域の契約を確認する。
既存スキルを更新する場合は、現在の記述、期待された挙動、実際の挙動を分けて確認する。

#### スキルとして残す判断を選ぶ
スキルには、複数の利用場面で再利用する方針、責務境界、作業契約、専門知識、ツール操作を残す。
個別事例から規則を導く場合は、事例の語彙ではなく、同じ判断を必要とする条件と不変の意図を抽出する。
一般的な推論だけで安定して導ける内容より、利用者や製品が所有する判断、実行順序が重要な作業、誤ると回復が難しい操作を優先する。

### Phase 2: 責務と発火面を設計する
#### 責務を固定する
スキルが受け取る状況、行う判断、残す成果、完了状態を一文で説明できるようにする。
一つのスキルには、一つの作業単位または一つの成果物に結び付く責務を持たせる。
単体で意味のある成果へ到達し、利用主体や上位フローを前提にせず完了できる構造にする。

#### 隣接スキルとの境界を確認する
既存スキルを名前だけでなく、入力、判断、成果、完了状態から比較する。
同じ判断と成果を持つ場合は既存スキルへ統合し、独立して利用できる責務がある場合は分ける。
下位手順を分ける場合は、その手順だけでも利用できる責務と成果があることを確認する。

#### 発火面を書く
`description` には、何を扱い、どのような状況で使い、何を成立させるかを書く。
自然な依頼から選択できる語彙を使い、本文を読まなければ発火条件が分からない状態にしない。
隣接スキルとの差は、他スキルの説明ではなく、このスキルが所有する対象と成果で表す。

### Phase 3: 判断と構造を設計する
#### 自由度を決める

| 作業の性質 | 記述方法 |
| --- | --- |
| 文脈によって複数の妥当な判断がある。 | 判断基準と優先順位を示す。 |
| 推奨する進行があり、状況に応じた分岐がある。 | フロー、分岐条件、戻り先を示す。 |
| 順序や出力が壊れやすく、決定論的に処理できる。 | 実行可能なスクリプトまたは厳密な操作契約を使う。 |

順序、反復、停止条件が成果に影響する作業はフェーズで構造化する。
順序を必要としない規則は、判断対象ごとの節にまとめる。
すべてを同じ書式へ押し込まず、判断の性質に合う構造を選ぶ。

#### 本文と補助ファイルを分ける
発火後の作業全体で使う目的と判断基準は本文に置く。
順序や反復が成果に影響する場合はフローを、成立状態を明示する必要がある場合は完了条件を本文に置く。
本文はスキルの発火後に読み込まれるため、一般知識の説明ではなく、そのスキルでなければ安定しない判断と行動にコンテキストを使う。
特定の条件でだけ使う詳細、長い規則、スキーマは `references` に分ける。
繰り返し実装される処理や、同じ結果を厳密に再現する必要がある処理は `scripts` に置く。
成果物の素材として使うファイルは `assets` に置く。
本文と補助ファイルの間で同じ規則を正本として重複させない。

### Phase 4: スキルを作成、更新、または評価する
作業種別に対応する節を使う。

#### 作成または更新する
`name` は短く、責務を識別できる名前にする。
本文はエージェントが作業中に必要とする判断と行動を直接記述する。
設計過程、会話の経緯、採用しなかった案は、現在有効な判断へ変換してから残す。
文章には `$writing` を適用し、同じ意味の言い換え、責務外の説明、不自然な強調を整理する。

更新では、次も行う。
既存の有効な契約を保持し、変更する判断と影響する箇所を対応させる。
新しい記述を加えた後は、重複した規則、古い入口、役割を失った参照を同じ変更で整理する。
局所的な修正後も、`description`、本文、補助ファイル、隣接スキルとの関係を通して読む。

#### レビューする
責務、発火面、判断の十分性、自由度、構造、補助ファイル、完了状態を評価する。
問題は、対象箇所、期待する挙動への影響、根拠、必要な差分が追える形で返す。

### Phase 5: 挙動を検証する
#### 検証を組み立てる
実際に想定される依頼と成果物を用い、作成時の文脈から切り離したエージェントでスキルを実行する。
検証は、共有状態を変更しない対象または隔離した複製で行う。外部状態への変更が必要な場合は、依頼で許可された範囲に従う。
検証するエージェントには、依頼の遂行に必要な情報だけを渡す。
期待する結論、疑っている問題、意図した修正を先に与えず、スキルから同じ判断へ到達できるかを見る。
作成時と切り離した文脈を用意できない場合は、実行できなかった挙動と理由を未検証範囲として残す。

#### 結果を評価する
次を確認する。

- 必要な状況でスキルが選択され、隣接する依頼へ不必要に発火しない。
- 責務内の判断を途中で省略せず、必要な成果と完了状態へ到達する。
- 具体例の表現を別の状況へ過剰適用していない。
- 本文と参照を必要な時点で読み、書かれていない前提を補っていない。
- 確認が必要な判断と、自律して進められる判断を区別している。

発火面に問題がある場合は Phase 2、判断や構造に問題がある場合は Phase 3、記述や補助ファイルに問題がある場合は Phase 4 に戻る。
修正後は、影響した挙動を新しい文脈で再検証する。

### Phase 6: 作業状態を残す
作成または更新では、完成したスキル、変更した判断、実施した挙動検証、確認が必要なことを残す。
レビューでは、成立している点、問題、根拠、必要な差分、未検証範囲を残す。
確認が必要なことには、決めるべきこと、今回はどう扱ったか、扱いによって変わる挙動を含める。

## 完了条件
- 責務、発火面、本文、補助ファイル、完了状態が一つの作業契約として整合している。
- 各規則に、再利用する判断または行動としての役割と根拠がある。
- 隣接スキルと責務が重複せず、単体で利用できる。
- 作成または更新では、挙動検証で意図した発火、判断、成果、完了状態が成立し、見つかった問題が残っていない。
- レビューでは、成立している点、問題、根拠、必要な差分、未検証範囲を返している。
