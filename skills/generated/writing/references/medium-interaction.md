# 対話とコメント

## 基本
- 相手が次に判断または実行できる形で返す。
- 結論を先に置き、必要な理由だけを続ける。
- 長い背景説明を、依頼、判断、結果の前に置かない。
- 相手に不要な作業過程を見せない。

## レビューコメント
- 対象、問題、理由、期待する変更を含める。
- 直せる指摘と、判断が必要な指摘を分ける。
- 好みの表現差だけを、設計上の問題のように扱わない。
- コード、仕様、文書のどれに対する指摘かを明確にする。

## Issue と PR の本文
- 読者が目的、変更範囲、完了条件、検証を判断できる構造にする。
- 作業ログや検討過程ではなく、残すべき決定と確認事項を書く。
- 変更理由、利用者への影響、検証結果を分ける。

## 返信
- 質問には、回答、理由、次の行動の順で返す。
- 未確認のことは未確認として扱う。
- 依頼を受けた場合は、実施内容と残件を分けて返す。
- 相手の指摘が正しい場合は、弁明より先に修正方針を示す。
- 誤りを認める場合も、作業ログではなく、何をどう直すかを書く。
